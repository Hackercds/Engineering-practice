# 审查双关：AI 自审 prompt + 合并前 checklist（企业级版）

> **"测试绿"和"可以上线"中间，差一步谁来担保。** 三道关：**机器自动拦**（CI 硬关）+ **AI 自审**（明显错）+ **人来批**（意图错 + 不可逆）。
>
> **本版本升级**：从"两道关"升级为"三道关 + 风险分级"——把不可逆/高风险操作显式建模，并接入企业级 PR 流程。

---

## 三道关模型

```
PR 提交
  ↓
[CI 硬关 · 自动]   测试 / 契约 / SAST / SCA / secrets / 覆盖率
  ↓ 过
[第一关 · AI 自审] 抓"明显错"（边界、异常、漏测、与 spec 偏差）
  ↓ 过
[第二关 · 人来批]  抓"和意图不符 + 不可逆 + 影响范围 + 业务语义"
  ↓ 过
合并
```

> **CI 硬关没过 → 哪关都不用审，直接打回。** 省人时间。

---

## 第零关 · CI 硬关（机器自动 · 不通过直接打回）

| 类别 | 工具（示例） | 拦什么 |
|---|---|---|
| 测试 | pytest / jest / go test | 功能正确性 |
| 覆盖率 | coverage.py / istanbul | 漏测 |
| 契约 | oasdiff / spectral | 改了契约不通知下游 |
| 契约 lint | spectral | 字段命名/描述/example |
| Lint / Format | eslint / ruff / golangci-lint | 风格 |
| 类型 | tsc / mypy / pyright | 类型 |
| SAST | bandit / semgrep / sonarqube | 注入 / 不安全 API |
| SCA | safety / snyk / dependabot | 依赖漏洞 |
| Secrets | gitleaks / trufflehog | 密钥泄露 |
| License | syft / licensee | 许可证合规 |
| SBOM | syft / cyclonedx | 物料清单 |
| IaC | tfsec / checkov | Terraform 风险 |
| Container | trivy / grype | 镜像漏洞 |
| Mutation | mutmut / stryker | 测的有效性 |

> **CI 硬关清单见** [07](07_CI门与发版策略.md)。**门槛值由团队/RFC 决定**，不在 PR 时临时拍。

## 第一关 · AI 自审 prompt（抓"明显错"）

> 复制这段作为系统 prompt 或单轮 prompt 都行。

```
你是严格的代码审查者。审查下面这段产出，**只列问题、不夸奖**。逐条检查：

1. 边界：空输入、极值、超长、并发、重复调用有没有处理？
2. 异常路径：失败时返回什么？有没有静默吞错？
3. 漏测：哪些行为没有对应测试？覆盖率是否下降？
4. 与 spec 的偏差：有没有做了 spec 没要求、或漏了 spec 要求的？
5. 可逆性：有没有不可逆副作用（删数据 / 动钱 / 对外发布）未加确认？
6. 安全：用户输入是否校验？SQL 是否参数化？secrets 是否泄露？
7. 可观测：关键路径是否有日志/指标？错误是否有错误码？
8. 性能：是否有明显 O(n²) / N+1 / 同步阻塞？

对每条给出：
- 问题（一句话）
- 它会在什么输入下出事
- 具体改法（可贴代码片段）
- 风险等级：低 / 中 / 高 / 不可逆
```

> **边界提醒**：AI 自审能抓"明显错"，**抓不住"代码对但理解错了需求"**——因为它不知道真实意图。所以还要第二关。

## 第二关 · 人来批 checklist（抓"和意图不符 + 不可逆"）

### 强制项（漏一项不合并）

- [ ] **意图**：这段实现真的解决了 issue 要的那个问题吗？（不是另一个相近问题）
- [ ] **可逆性**：有没有不可逆副作用？有 → 必须人确认 + 灰度 + feature flag
- [ ] **影响范围**：改这块会牵动谁？下游 / 调用方 / 数据格式需要同步改吗？
- [ ] **可回滚**：出问题能不能秒级回退？要不要 feature flag？

### 业务项（按需勾选）

- [ ] **业务语义**：对吗？（机器判不了的那部分——命名、字段含义、边界口径）
- [ ] **测试覆盖**：关键路径和边界都覆盖了吗？**故意改坏一处，测试会红吗？**
- [ ] **数据迁移**：有没有 schema 变更？是否向后兼容？降级路径？
- [ ] **文档同步**：README / ADR / Runbook / Changelog 更新了吗？
- [ ] **可观测**：新指标 / 告警 / 仪表盘加了吗？
- [ ] **法务 / 合规**：PII / GDPR / 出口管制 / 第三方授权 触线了吗？
- [ ] **客户沟通**：是否需要发公告 / changelog / 升级指南？

> **这两条最容易被跳过、也最致命**：意图错（解决错问题）+ 不可逆（出问题救不回来）。

---

## 风险分级与处理

| 风险等级 | 例子 | 处理 |
|---|---|---|
| **低** | 重构、加日志、改文案 | 1 人审 + CI 过即可合并 |
| **中** | 新增端点、依赖升级、配置变更 | 1 人审 + AI 自审 + CI 过 |
| **高** | 改 schema、动钱、对外 API 变更 | 2 人审（一人架构）+ AI 自审 + CI 过 + 灰度 + 监控 |
| **不可逆** | 删数据、发布、回滚不了的操作 | 2 人审 + 工单 + 变更窗口 + 演练过的回滚预案 + 现场值守 |

## PR Template（建议放进 `.github/PULL_REQUEST_TEMPLATE.md`）

```markdown
## 关联
- Spec: docs/spec/SPEC-XXX.md
- Issue: #XXX
- ADR: docs/adr/XXXX.md（如有）

## 动机
为什么改？改了什么问题？

## 改动
- 改动 1
- 改动 2

## 影响范围
- 影响的下游/调用方：
- 是否需要数据迁移：
- 是否需要客户端升级：

## 测试
- 单元 / 集成 / 契约 / E2E：
- 覆盖率：
- 故意改坏自检：✅ / ❌

## 风险与回滚
- 风险等级：低 / 中 / 高 / 不可逆
- 不可逆操作：（无 / 列出 + 确认人）
- 回滚方案：
- feature flag：

## 文档
- README / ADR / Runbook / Changelog：（已更新 / N/A）

## Checklist
- [ ] CI 硬关全过
- [ ] AI 自审 prompt 跑过
- [ ] 上方所有强制项已勾
- [ ] 至少 1 位 reviewer approve（高风险 2 位）
```

## 位置

把三道关都放在**合并前 / 上线前**，让风险进主干之前被拦住，而不是上线后救火。

---

## 进阶：让三关的成本更低

| 做法 | 收益 |
|---|---|
| AI 自审 prompt 写进 `AGENTS.md` / `.claude/commands/` | 每次 PR 自动跑，不靠人记 |
| Checklist 做成 PR template | 提交时强制勾选，漏一项 CI 红 |
| 契约 diff / lint 进 CI（oasdiff / spectral） | 改 API 时下游立刻见红 |
| 故意改坏一处验证测试 | 一次性动作，验证"测试真的在兜底" |
| 不可逆操作加 feature flag / dry-run / 二次确认 | 出问题秒回退 |
| Mutation testing 进 CI | 验证"测试有效"而非"测试存在" |
| Reviewer 自动分配（CODEOWNERS） | 高风险文件自动找对的人 |

## 一句话判断

> **CI 自动拦 + AI 自审 + 人来批，三道关都过了，再合并**——这是把"AI 写的代码信不过"变成"信得过"的企业级闭环。
