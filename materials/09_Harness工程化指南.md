# Harness 工程化指南（企业级 · 新增）

> **Harness Engineering = 围绕 AI Coding Agent 搭建的"约束 + 反馈 + 验证"系统**。
> 把 AI 当马，Harness 就是缰绳 + 路线 + 围栏 + 反馈——2026 年的 AI 工程化核心范式。
>
> 本卡把 Harness 拆为 **9 维**，每维给目标、做法、工具、度量。读完即可上手搭一套。

---

## 0. 一图总览

```
        Coding Agent = Model + Context + Tools + Constraints + Feedback + Human Control
                       ↑         ↑        ↑          ↑            ↑            ↑
                       9 维     1 维     2 维       3 维        4 维          5 维
                                                                          （+ 6-9 维配套）
```

> 选完模型只是起点；**6/7 维以后的工程化才是分水岭**。

## 1. 维度 1 · Model（基座）

> 不是重点——同一模型在不同 Harness 下表现天差地别（LangChain 实验：同一模型 + 优化 Harness，从 30+ 冲到 Top 5）。

| 关注点 | 做法 |
|---|---|
| 选型 | 见 [04](04_coding-agent选型对照表.md) |
| 升级 | 不拍脑袋——走 eval 回放（[08](08_AgentOps_可观测与回滚.md)） |
| 私有部署 | 涉密场景走本地 / VPC 部署（Junie / 自托管） |
| 多模型路由 | 简单任务用小模型，复杂任务用大模型，省成本 |

## 2. 维度 2 · Context（上下文）

> AI 看到的"世界"——给得对，它就懂；给得乱，它就猜。

### 2.1 必给的上下文

| 类型 | 来源 | 工具 |
|---|---|---|
| **AGENTS.md / CLAUDE.md** | 工程宪法 | 手动维护 |
| **当前 Spec** | SPEC-XXX.md | CI 自动注入 |
| **当前任务上下文** | issue / PR / diff | agent 自动拉 |
| **代码库索引** | 文件、符号、依赖 | Cursor / Cody / Sourcegraph |
| **相关历史** | 相似 PR / 相似 issue | 检索 / 摘要 |
| **运行时反馈** | 测试结果 / 错误日志 / trace | agent 自动拉 |

### 2.2 不要塞进上下文的

- 整个 monorepo（用 RAG / 索引）
- 上次 session 的对话历史（除非显式带入）
- 不相关 spec / 文档

### 2.3 上下文优化技巧

- 摘要旧对话，保留关键事实
- 工具调用结果只回显关键字段
- 长代码 → 切片 + 行号引用

---

## 3. 维度 3 · Tools（工具）

> 工具是 agent 的"手"——能调的工具越多，能做的事越多。

### 3.1 必给工具

| 工具 | 用途 | 标准 |
|---|---|---|
| **read_file / write_file / edit_file** | 文件 I/O | 必备 |
| **bash** | 跑命令 | 必备，但需沙箱 |
| **grep / glob** | 检索 | 必备 |
| **git** | 版本控制 | 必备 |
| **跑测试** | pytest / jest | 必备 |
| **查错误** | 看 log / stack | 必备 |
| **查 spec / ADR** | 看 docs | 强烈建议 |
| **创建 PR** | gh CLI | 强烈建议 |
| **查 ticket** | Jira / Linear | 建议 |
| **查监控** | Datadog / Grafana | 高级 |

### 3.2 工具协议

| 协议 | 场景 |
|---|---|
| **MCP（Model Context Protocol）** | 跨工具统一接口，企业级首选 |
| **OpenAI Function Calling** | OpenAI 生态 |
| **Anthropic Tool Use** | Claude 生态 |
| **LangChain Tools** | LangChain 生态 |

> **优先用 MCP**——它是企业级标准，便于跨 agent 复用工具。

### 3.3 工具沙箱

> **不能让 agent 直接动 prod**——所有写操作默认沙箱内。

| 沙箱级别 | 场景 |
|---|---|
| **dry-run** | 只读 + 模拟写 |
| **staging** | 写到测试环境 |
| **feature branch** | 写到分支，需 PR 合并 |
| **prod（受限）** | 仅限白名单操作 + 二次确认 |

## 4. 维度 4 · Constraints（约束 · 核心）

> **Harness 的灵魂**——把 AI 自由发挥的空间一层层夹紧。

### 4.1 约束的 4 个层次

```
┌─────────────────────────────────────────┐
│  4. 法律 / 合规（不可破）               │  ← PII / GDPR / 出口管制
├─────────────────────────────────────────┤
│  3. 安全（不可破，硬关）                │  ← secrets / 注入 / 越权
├─────────────────────────────────────────┤
│  2. 架构 / 契约（CI 守）                │  ← OpenAPI / 模块依赖 / 命名
├─────────────────────────────────────────┤
│  1. 风格 / 口味（lint 守）              │  ← AGENTS.md / lint / format
└─────────────────────────────────────────┘
```

> **约束越具体越好**。"代码要优雅"（不可执行）vs "函数不超过 50 行 / 嵌套不超过 3 层"（可 lint）。

### 4.2 约束的实现位置

| 位置 | 拦什么 |
|---|---|
| **AGENTS.md** | 软约束（AI 主动遵守） |
| **Lint / Format** | 风格（自动修） |
| **类型检查** | 类型（编译期） |
| **测试 / 覆盖率** | 功能 + 漏测（CI 硬关） |
| **契约 diff / lint** | API 兼容（CI 硬关） |
| **SAST / SCA / secrets** | 安全（CI 硬关） |
| **架构 lint** | 模块依赖 / 分层（CI 硬关） |
| **预提交 hook** | 提交前最后一道（husky / pre-commit） |
| **运行时策略** | prod 操作白名单 / 二次确认 |

### 4.3 约束的演进

- 跑偏一次 → 补一条约束
- 约束 > 20 条 → 拆子目录 AGENTS.md
- 约束冲突 → 显式分级（安全 > 架构 > 风格）

## 5. 维度 5 · Feedback（反馈 · 核心）

> **Harness 的另一半灵魂**——AI 不知道对错，必须有反馈。

### 5.1 反馈的 4 类来源

| 来源 | 速度 | 价值 |
|---|---|---|
| **测试结果** | 秒级 | 极高（机器能判的合同） |
| **Lint / 类型** | 秒级 | 高（自动修） |
| **AI 自审** | 10s 级 | 中（抓明显错） |
| **人审** | 小时–天 | 极高（抓意图错） |

### 5.2 反馈的形式

| 形式 | 例子 |
|---|---|
| **结构化** | `{passed: true, fail_count: 0, errors: [...]}` |
| **diff** | "改了 3 个文件，新增 50 行" |
| **可读摘要** | "test_blank_title_rejected_422 红了，输入 `'   '` 期望 422" |
| **修复建议** | "建议在 `validate_title` 中加 `if not title.strip(): raise 422`" |

### 5.3 反馈的"颗粒度"

> **反馈越具体，AI 越能修**。失败信息要带：断言 + 输入 + 期望 + 上下文。

- ❌ "有 bug"
- ❌ "测试失败"
- ✅ "test_blank_title_rejected_422 红了：输入 `'   '` 期望 422，实际返回 200。原因：`validate_title` 没去空白就校验。"

## 6. 维度 6 · Human Control（人控）

> **AI 写得快，但人拍板**——这是不可让渡的。

### 6.1 人控的 4 个位置

| 位置 | 人做什么 |
|---|---|
| **意图** | 写 Spec、划范围、定不做 |
| **审查** | 审 PR、抓意图错、批不可逆 |
| **不可逆** | 删数据 / 动钱 / 发布 / 改线上配置——人确认 |
| **事故** | 回滚 / 沟通 / 复盘 |

### 6.2 人控的"轻量化"

- 模板化（PR template / Spec template）→ 减少填写成本
- 自动化（CI 硬关 / AgentOps）→ 减少评审成本
- 风险分级（见 [06](06_审查双关_prompt与checklist.md)）→ 低风险轻审、高风险重审

### 6.3 "人在回路" 的几种模式

| 模式 | 场景 |
|---|---|
| **Human-on-the-loop**（监控中） | 低风险任务：AI 干，人看着告警 |
| **Human-in-the-loop**（必经） | 中风险：AI 干完人审 |
| **Human-in-command**（人主导） | 高风险 / 不可逆：人下指令，AI 执行 |

## 7. 维度 7 · Memory（记忆）

> **AI 跨 session 不记事**——靠"持久化"补。

| 层级 | 持久化什么 | 工具 |
|---|---|---|
| **项目级** | AGENTS.md / Spec / ADR / Runbook | git |
| **会话级** | 当前 session 上下文 | 工具自带 |
| **跨会话** | 团队知识库 / 决策记录 | Notion / Confluence / ADRs |
| **Agent 级** | 偏好 / 风格微调 | Cursor rules / CC memory |
| **Eval 级** | 金标集 / 历史评估 | Langfuse / Braintrust |

> **记忆的污染防护**：过期信息要明确标"已废弃"，避免 AI 拿到错上下文。

---

## 8. 维度 8 · Eval（评估）

> **AI 升级、模型切换、prompt 改动——怎么知道变好变坏？靠 Eval。**

见 [08 §维度 2](08_AgentOps_可观测与回滚.md)。这里强调：

| 关键 | 做法 |
|---|---|
| **金标集** | 手工/脚本构造，> 30 条，覆盖核心场景 |
| **回放** | 模型升级 / prompt 改动后跑同一金标集 |
| **趋势** | Eval 分数进监控，长期看趋势 |
| **分级** | 重大变更（金标集变化大）→ 全面 Eval；小改动 → 抽样 |
| **人评** | Eval 不替代人评，复杂场景加 human review |

| 维度 | 必做（Week 1） | 应做（Month 1） | 长期（Quarter） |
|---|---|---|---|
| 1. Model | 选定主模型 + 备选 | 多模型路由 | 模型升级走回放 |

| 3. Tools | 必备 5 件套 + 沙箱 | MCP 工具集 | 工具权限审计 |
| 4. Constraints | AGENTS.md + 基础 lint | CI 12 门 | 架构 lint / 运行时策略 |
| 5. Feedback | 测试 + lint 反馈回路 | AI 自审 + 修复建议 | 反馈自动生成 PR fix |
| 6. Human Control | PR template + 风险分级 | 不可逆审批流 | 团队 AI 协作 SOP |
| 7. Memory | AGENTS.md + Spec | ADR / Runbook | 跨项目知识图谱 |
| 8. Eval | 建金标集（≥ 30） | Eval 跑分进 PR | 模型升级走回放 |

## 11. 常见坑

| 坑 | 后果 | 改成什么 |
|---|---|---|
| 工具 > 100 个 | agent 选错 / 拖慢 | 工具 < 20 个，按场景装 |
| 约束太死 | AI 啥也干不了 | 分级：硬约束 + 软约束 |
| 反馈糊 | AI 改不动 | 反馈结构化 + 具体 |
| 人控全在 PR | 评审变瓶颈 | 前置：Spec / CI 硬关 / AgentOps |
| Eval 凭感觉 | 升级翻车 | 金标集 + 量化 |
| 记忆无清理 | agent 拿到过期信息 | 明确"已废弃"标识 |
| Harness 一次搭完 | 没人维护 / 跑偏 | 跑偏一次补一条，渐进完善 |
| 模型当重点 | 忽略约束 + 反馈 | 模型只是 1/9 |

---

## 12. 一句话心法

> **Coding Agent = Model + Context + Tools + Constraints + Feedback + Human Control。**
> **Harness 的 9 维 = 把 AI 从"提示词炼丹"升级为"工程化产线"。**
> 一次搭好 Harness，团队复用，规模化的能力 = 模型 × Harness，不是模型本身。
