# CI 门与发版策略（企业级 · 新增）

> **CI 硬关是 AI 时代工程化的"地基"**——没有它，所有"质量保证"都是人肉活。
> 本卡定义：分支策略、CI 必装门、PR 流程、保护规则、发版策略、回滚预案。

---

## 1. 分支策略（推荐：trunk-based + 短命分支）

```
main (受保护)
  │
  ├── feat/SPEC-001-post-todos      ← 短命（< 2 天）→ squash merge → 删除
  ├── fix/SPEC-001-id-404
  ├── chore/bump-deps
  └── release/v1.2.0                ← 标签 + 发版
```

| 规则 | 强制 |
|---|---|
| main 受保护，必须 PR 合并 | ✅ |
| 必须 squash merge（commit 历史干净） | ✅ |
| 必须删除已合并分支 | ✅ |
| 禁止 force push to main | ✅ |
| 禁止直接 push to main | ✅ |
| hotfix 可从 main → fix/ → main（CI 全跑） | ✅ |
| 长期 feature 分支（> 2 周）= 反模式 | ❌ |

## 2. CI 必装门（企业级 12 门）

> **门槛值由团队 RFC 决定**，不要在 PR 时临时拍板。

| # | 门 | 工具（示例） | 拦什么 | 失败时 |
|---|---|---|---|---|
| 1 | **单元 + 集成测试** | pytest / jest | 功能正确性 | block merge |
| 2 | **覆盖率门槛** | coverage.py / istanbul | 漏测 | block merge（降覆盖需说明） |
| 3 | **契约测试 / diff / lint** | oasdiff / spectral / dredd | 改 API 不通知 | block merge |
| 4 | **E2E（关键旅程）** | playwright / cypress | 用户视角 | block merge（main 必跑） |
| 5 | **Lint / Format** | eslint / ruff / golangci-lint | 风格 | block merge |
| 6 | **类型检查** | tsc / mypy / pyright | 类型 | block merge |
| 7 | **SAST** | bandit / semgrep / sonarqube | 注入 / 不安全 API | block merge（高危）/ warn（低危） |
| 8 | **SCA** | safety / snyk / dependabot | 依赖漏洞 | block merge（高危 CVE）/ warn |
| 9 | **Secrets 扫描** | gitleaks / trufflehog | 密钥泄露 | **block merge（绝不放过）** |
| 10 | **License / SBOM** | syft / licensee | 合规 | warn / block（看团队策略） |
| 11 | **IaC 扫描** | tfsec / checkov | Terraform 风险 | block merge（高危） |
| 12 | **容器镜像扫描** | trivy / grype | 镜像漏洞 | block deploy（高危 CVE） |

> **第 9 门（secrets）和第 7/8 门高危漏洞 = 绝不放行**。其他可设 warn→block 渐进。

## 3. PR 流程（与 [06](06_审查双关_prompt与checklist.md) 衔接）

```
feat/SPEC-001-post-todos
  ↓
[Push]  → CI 全跑 → 12 门过 / 失败
  ↓ 过
[Open PR] → PR template 必填 → AI 自审 comment（自动）
  ↓
[Reviewer]  → 按 [06 §第二关](06_审查双关_prompt与checklist.md) 勾选
  ↓ approve
[CODEOWNERS]  → 高风险文件自动找对应 reviewer
  ↓ approve
[Merge]  → squash → 删除分支
```

### 3.1 PR Template 强制项（与 [06](06_审查双关_prompt与checklist.md) 对齐）

- 关联 Spec
- 动机 / 改动 / 影响范围
- 风险等级 + 回滚方案
- 不可逆操作清单 + 确认人
- 文档更新（README / ADR / Runbook / Changelog）

### 3.2 CODEOWNERS（自动路由 reviewer）

```gitignore
# .github/CODEOWNERS
/spec/                 @team-arch
/backend/auth/         @team-security
/infra/                @team-platform
/AGENTS.md             @team-arch
```

## 4. 保护规则（main 分支）

| 规则 | 推荐设置 |
|---|---|
| Require PR before merging | ✅ |
| Require approvals | 1（低风险）/ 2（高风险 / 不可逆） |
| Dismiss stale approvals on push | ✅ |
| Require review from Code Owners | ✅ |
| Require status checks to pass | ✅ 列出所有必过门 |
| Require up-to-date branches | ✅ |
| Require conversation resolution | ✅ |
| Require linear history | ✅ |
| Require signed commits | ✅（高敏场景） |
| Include administrators | ✅（防自己绕过） |
| Restrict who can push | 仅 release bot |
| Allow force pushes | ❌ |
| Allow deletions | ❌ |

## 5. 发版策略

### 5.1 语义化版本（SemVer）

```
v<major>.<minor>.<patch>
       ↑          ↑        ↑
       破坏变更   新增功能   bug 修复
```

- 改 Public API 契约 → 必须 major
- 新增端点 / 字段（向后兼容）→ minor
- bug 修复 / 内部重构 → patch

### 5.2 标签与发版

```bash
# 自动化
git tag v1.2.0
git push --tags
# CI 触发：build → 镜像扫描 → 签名 → 推 registry → 部署 staging → 跑 E2E → 手动 promote prod
```

### 5.3 变更日志（自动生成）

| 工具 | 来源 |
|---|---|
| release-please / semantic-release | Conventional Commits |
| git-cliff | Conventional Commits + 自定义模板 |
| 手写 | 不推荐（必然漏） |

### 5.4 灰度发布（企业级必备）

| 策略 | 工具 | 适用 |
|---|---|---|
| Feature flag | LaunchDarkly / Unleash / 自建 | 全场景 |
| 蓝绿 | K8s / ALB | 无状态服务 |
| 金丝雀 | Istio / Argo Rollouts | 有 SLO 要求 |
| 影子流量 | mesh mirroring | 高风险变更验证 |

> **不可逆操作必须 feature flag + 金丝雀**。

## 6. 回滚预案（必备）

| 准备项 | 检查 |
|---|---|
| 数据库迁移可回滚 | 每次 migration 配套 down |
| 前后端 API 兼容 | 新字段可选 / 新端点非破坏 |
| 镜像可秒回上一版 | K8s `kubectl rollout undo` / registry tag 保留 |
| 配置变更可还原 | GitOps / IaC 一键回滚 |
| Feature flag 可秒关 | 不需要重新部署 |
| 数据可恢复 | 备份策略 + 恢复演练 |
| 客户可降级 | 旧客户端能跑（API 兼容） |

> **回滚预案必须在 PR 阶段写好**——出问题时现想来不及。

---

## 7. 变更窗口（高风险 / 不可逆）

| 类型 | 窗口 |
|---|---|
| 日常低风险 | 任意工作时间 |
| 中风险 | 业务低峰 + 监控就位 |
| 高风险 / 不可逆 | 提前公告 + 业务低峰 + 双人值守 + 演练过的回滚 |
| 跨时区业务 | 全球低峰（如 UTC 03:00） |

## 8. CI 工作流骨架（GitHub Actions 示例）

```yaml
# .github/workflows/ci.yml
name: CI
on:
  pull_request:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.10' }
      - run: pip install -r requirements.txt -r requirements-dev.txt
      - run: pytest --cov=app --cov-fail-under=80
      - run: oasdiff diff origin/main HEAD --format github
      - run: spectral lint openapi.yaml
      - run: bandit -r app/
      - run: safety check
      - run: gitleaks detect --redact
      - run: mutmut run  # mutation testing（按需，PR 体积大时跳过）
```

> 完整工作流按团队栈选——本节只给骨架。

## 9. 反模式（看到就停）

| 反模式 | 后果 | 改成什么 |
|---|---|---|
| 直接 push main | 绕过所有质量门 | PR + 保护规则 |
| "CI 太慢先合再说" | 烂代码污染主干 | CI 必装项瘦身并行化 |
| 长期 feature 分支 | 合并地狱 | trunk-based + 短命分支 |
| 改完 prod 再回滚 | 用户在用才救 | 灰度 + flag + 监控 |
| 没有回滚预案 | 出事瞎救 | PR 必填回滚方案 |
| 手动 bump 版本 | 漏 / 错 | 自动 release-please |
| 跳过 SAST "误报太多" | 真问题被埋 | 调规则 + 单 issue 跟踪 |
| 强制 push to main | 丢历史 / 难回溯 | 禁 force push |
| secrets 在 .env 入库 | 全网泄露 | secret manager + 预提交扫描 |
