# Spec 模板（SDD 用 · 升级版）

> **动手让 AI 写代码前，先和它一起把这张表填出来。** 你审一遍、改两处，再让它写。
> 你审的是这张表（十几行），不是它写的实现（上百行）。
>
> **本版本升级**：在原 Spec 基础上增加"实现计划 / 任务清单 / 验收标准"三节，让 Spec 同时驱动开发、测试、CI。

---

## 0. 元信息（让 AI / 后人知道这份 Spec 是什么）

| 字段 | 值 |
|---|---|
| 编号 | SPEC-YYYYMMDD-NNN（如 `SPEC-20260607-001`） |
| 名称 | <一句话：本 Spec 描述什么能力> |
| 版本 | v0.1（草案）/ v0.9（评审中）/ v1.0（已确认） |
| 负责人 | <谁最后拍板> |
| 关联 issue / 需求 | <链接> |
| 关联 ADR | <架构决策记录链接，如有> |
| 状态 | 草案 / 评审中 / 已确认 / 已实现 / 已废弃 |

---

## 1. 一句话目标

> 这个东西要解决什么、做成什么样算对。
> **不超过两行**。写不出来说明还没想清楚——回去和用户聊。

---

## 2. 范围

### 2.1 必须有（In-scope）
- 功能 1
- 功能 2

### 2.2 明确不做（Out-of-scope · 防止 AI 自由发挥）
- 不做 XX、不做 YY、不做持久化、不做鉴权……
> **这一节比"做什么"还重要**。AI 默认会做你没禁止的事——把"不做"列清楚。

---

## 3. 数据模型

| 字段 | 类型 | 约束（必填？范围？默认？） | 示例 |
|---|---|---|---|
|  |  |  |  |

> **约束要写到能写成测试的程度**："非空"不够，要写"去除首尾空白后非空"；"长度合理"不够，要写"1–100"。

---

## 4. 端点 / 接口

| 动作 | 输入 | 成功返回 | 失败（状态码 + 触发条件） |
|---|---|---|---|
|  |  |  |  |

> **状态码不一致是联调灾难**。每条失败要给出**触发条件**和**状态码**。

---

## 5. 边界（每条之后会变成一条测试）

- 空 / 极值 / 重复 / 不存在 / 冲突 / 并发 / 超长 / 国际化 / 时区 / 编码 ……
- 写"边界"时想象一个想搞事的用户会怎么用。

---

## 6. 验收标准（Definition of Done）

- [ ] 所有端点 happy path 测试通过
- [ ] Spec §5 边界每条对应 1 条测试
- [ ] 契约（OpenAPI/typed client）已导出
- [ ] 关键指标（延迟/错误率/资源占用）有基线
- [ ] 安全门（SAST/SCA/secrets）通过
- [ ] 文档/ADR/变更日志已更新

---

## 7. 实现计划（从 Spec 派生的任务分解）

> 一个端点一个 TDD 循环，不要一次让 AI 写完整个服务。

---

## 8. 任务清单（可逐项验收的小块）

> 这一节是**给 CI 看的"工单"**——每条可独立开 PR、独立审查、独立合入。

- [ ] T1：搭建脚手架（脚手架命令见 AGENTS.md）
- [ ] T2：数据模型 + 校验
- [ ] T3：端点 1（先红再绿）
- [ ] T4：端点 2（先红再绿）
- [ ] T5：端点 3（先红再绿）
- [ ] T6：契约导出
- [ ] T7：审查双关 + 合并

---

## 9. 非功能性需求（按需启用）

| 维度 | 指标 | 备注 |
|---|---|---|
| 性能 | P95 < Xms | 压测脚本：… |
| 可用性 | X 个 9 | SLO：… |

## 自检（3 条，Spec 评审前先过一遍）

- [ ] **每条规则能不能写成测试？** 不能就是没说清。
- [ ] **人能在 2 分钟内审完吗？** 不能就拆 Spec。
- [ ] **范围划死了吗？** §2.2 有东西吗？空的就是没划。

## 填好的样子（参考）

```markdown
## 0. 元信息
- 编号：SPEC-20260607-001
- 名称：待办事项 API
- 版本：v1.0（已确认）
- 状态：已确认

## 1. 一句话目标
对外提供 Todo 的增 / 查 / 标记完成；内存存储，3 个端点。

## 2.1 必须有
- POST /todos（新建）
- GET  /todos（列表）
- POST /todos/{id}/done（标记完成，幂等）

## 2.2 明确不做
- 不做删除、不做编辑、不做分页、不做持久化、不做鉴权

## 3. 数据模型 Todo
| 字段 | 类型 | 约束 | 示例 |
|---|---|---|---|
| id   | int  | 服务端自增，从 1 开始，只读 | 1 |
| title| str  | 必填，去除首尾空白后非空，长度 1–100 | "买菜" |
| done | bool | 默认 false | false |

## 4. 端点
| 动作 | 输入 | 成功返回 | 失败 |
|---|---|---|---|
| POST /todos | {title: str} | 201 + Todo | 422（空/超长）|
| GET  /todos | — | 200 + Todo[] | — |
| POST /todos/{id}/done | — | 200 + Todo | 404（id 不存在）|

## 5. 边界
- 空标题 "" / 仅空白 "   " → 422
- 超长 101 字 → 422
- 标记不存在的 id → 404
- 重复标记完成：幂等，200

## 6. 验收
- [ ] 8 条测试全绿
- [ ] openapi.json 已导出
- [ ] SAST/SCA/secrets 通过

## 7. 计划
1. T1 脚手架 → 2. T2 数据模型 → 3. T3-5 三端点（每端点红→绿）→ 4. T6 契约 → 5. T7 审查

## 8. 任务
- [ ] T1: 脚手架
- [ ] T2: 数据模型
- [ ] T3: POST /todos
- [ ] T4: GET  /todos
- [ ] T5: POST /todos/{id}/done
- [ ] T6: openapi.json
- [ ] T7: 审查 + 合并
```

## 一句话收束

> **Spec 是下游所有东西的源头**：计划、任务、测试、实现、契约、文档、ADR 都是派生物。
> 改实现前先改 Spec——Spec 是和人对齐的载体，不是事后文档。
