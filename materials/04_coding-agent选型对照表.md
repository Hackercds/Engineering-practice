> **选型第一轴 = 对 spec 的遵从度**（这正是 SDD 这门课押注的东西），再叠加其余维度。

---

## 2. 一句话定位

- **Codex**：复杂任务、强 spec 遵从场景的首选体感。
- **Claude Code 原厂**：自主探索强、UI 审美好、文科类任务好。
- **CC + DeepSeek**：spec 遵从略有偏差、耗时长，但整体交付够用、**便宜**、**中文文案明显更顺**——性价比路线。
- **Cursor**：IDE 内联体验最好，UI 审美强；spec 遵从略弱于 Codex/CC 原厂。
- **JetBrains Junie**：JetBrains 全家桶用户的最佳搭档；**企业部署**有优势。
- **GitHub Copilot**：早期低端平替，**现不推荐**——Pro 对大陆新用户有限制、试用政策反复。

---

## 3. 怎么用这张表

按你自己的场景给维度加权重：

| 你的场景 | 优先选 |
|---|---|
| 强 spec / 复杂任务 | Codex / Claude Code 原厂 |
| 中文文案多、预算紧 | CC + DeepSeek |
| 要自主探索和审美 | CC 原厂 / Cursor |
| 团队强 JetBrains 习惯 | Junie |
| 企业合规 / 私有部署 | Junie / 自托管栈 |
| 简单补全 / 行内补全 | 任意一个 |

> 政策类信息（Copilot / Cursor 计费）最易过期，**用前自己再确认一次**。

---

## 4. 选型心法（比表更重要）

1. **没有"最好"，只有"最匹配当前任务"**——同一项目里后端用 A、前端用 B 很正常。
2. **把工具当"干活的"，不当"评判的"**——spec + 测试 + 审查才是质量的真正来源。
3. **跑偏一次就回来看这张表**——是 spec 没写清、还是工具真不行？先别换工具，先把 spec 改紧。
4. **Rules（`AGENTS.md` / `CLAUDE.md`）跨工具生效**——换工具不丢口味。
5. **企业级 ≠ 用最贵的**——Harness 搭好，工具只是手感差异；Harness 缺失，神仙工具也救不了。

---

## 5. 进阶：多 agent 协同（企业级实践）

| 角色 | 推荐 | 备注 |
|---|---|---|
| 主写代码 | Codex / CC 原厂 | spec 遵从优先 |
| 写中文文案 / 文档 | CC + DeepSeek | 中文顺、成本低 |
| Code Review | 任意一个 + 审查 prompt | 工具不是关键，prompt 才是 |
| 探索 / brainstorm | CC 原厂 | 自主探索强 |
| 改一行 / 行内补全 | Copilot / Cursor | 不必上重型 agent |
| 安全扫描 | SAST 工具（不进 agent） | 工具是工具，agent 是 agent |
| 契约 diff 评审 | CI 自动（Oasdiff） | 不必用 agent |

> **不要所有事都让 agent 干**。安全/合规/部署/审计走专用工具，agent 专注"生成"。

## 6. 选型 × Harness 9 维（交叉评估）

> 选完工具后，回到 [09](09_Harness工程化指南.md) 检查 9 维设计是否对齐。

| 维度 | 工具选型的反作用 |
|---|---|
| Context | 选上下文大的（CC、Codex）能装下更复杂的 spec |
| Tools | 选 MCP 生态好的（CC、Cursor）能接更多内部工具 |
| Constraints | 工具本身是否支持 `AGENTS.md`（多数已支持） |
| Feedback | 工具是否能输出结构化产物（diff、test result） |
| Human Control | 工具是否支持 plan-then-act（CC 原厂 / Codex 强） |

---

## 一句话判断

> **选型是杠杆，Harness + Spec 才是支点。** Spec 写得差，神仙工具也救不了；Harness 搭得紧，工具只是手感差异。
