# 测试先行清单（TDD · 企业级升级版）

> **让 AI 写实现前，先让它（按 spec）写测试。** 测试是 AI 的"退出标准"——红转绿才算完。
>
> **本版本升级**：在原红→绿→重构基础上增加 **契约测试 / 性能基线 / 故意改坏自检 / 覆盖率门槛**，让测试成为 CI 硬关。

---

## 0. 节奏：红 → 绿 → 重构，一个循环

```
Spec（已确认）
  ↓
[红] 按 spec 写测试 → 跑 → 应该全红（没实现）
  ↓
[绿] 让 AI 补实现 → 跑到全绿
  ↓           ↳ 中途有红？把"具体断言 + 输入"喂回去，别只说"有 bug"
[重构] 测试保持绿的前提下清理实现
```

> **一个端点一个循环**。不要一次让 AI 写完整个服务——这是"渐进式复杂度管理"的核心。

---

## 1. 测试金字塔（企业级必备）

> 不同层级测试覆盖不同范围，**不要全堆在单元**。

```
        ╱  E2E  ╲         少而精：关键用户旅程（5-10 条）
       ╱─────────╲
      ╱  契约测试  ╲      端点对外承诺：每个端点 ≥1（自动从 OpenAPI 派生）
     ╱───────────────╲
    ╱   集成 / 组件    ╲   模块协作：DB、缓存、消息、外部 API
   ╱─────────────────────╲
  ╱        单元           ╲  函数 / 类级别：边界、错误路径、纯逻辑
 ╱───────────────────────────╲
```

| 层级 | 谁写 | 速度 | 数量建议 |
|---|---|---|---|
| 单元 | AI 按 Spec 写 | < 10ms / 条 | 大量 |
| 集成 | AI 写 + 人审 schema | < 1s / 条 | 中等 |
| 契约 | **自动从 OpenAPI 派生** | < 500ms / 条 | 每端点 1+ |
| E2E | 人写关键旅程 | < 30s / 条 | 5–10 |

---

## 2. 红阶段 · 写测试

- [ ] **正常路径**：每个端点至少 1 条 happy path 测试
- [ ] **spec §5 边界小节每条 1 条**：空 / 极值 / 重复 / 不存在 / 冲突 / 并发 / 超长 ……
- [ ] **断言要具体**：期望值、状态码、关键字段都要写；不要只 `assert response.ok`
- [ ] **契约测试自动派生**（Pact / Spectral / Dredd）——从 OpenAPI 生成
- [ ] **错误路径**：每个端点的每个失败码至少 1 条
- [ ] **权限/角色**（如有）：未授权 / 越权 各 1 条
- [ ] 此时**没有实现，应该全红**——红得不彻底说明测试没在兜底

---

## 3. 绿阶段 · 补实现

- [ ] 让 AI 补实现，跑到**全绿**
- [ ] **遇到红**：把**失败的具体断言 + 输入**喂回去
  - ✅ `test_blank_title_rejected_422` 红了，输入 `"   "`，期望 422
  - ❌ "列表接口有 bug"
- [ ] **不许"为通过"改测试**；测试改了，spec 也要跟着改（见 [01](01_spec模板.md)）
- [ ] **不许"为通过"加 sleep / mock 掉失败分支**——这是慢性自杀

---

## 4. 重构阶段 · 清理实现

- [ ] 测试保持全绿
- [ ] 让 AI 在不改外部行为的前提下清理实现（命名、抽公共函数、去重复）
- [ ] 重构后再跑一次测试，确认没破

---

## 5. 验证"测试真的在兜底"（必做）

> **故意改坏一处实现 → 对应测试应该立刻变红。**
> 变不红，说明这条没被测住——把测试写得更紧。

这步比"测试通过"还重要：它是**测试有效性的自检**。

```
故意改坏 1 处（如：把空标题校验去掉）
  ↓
跑测试
  ↓
至少 1 条测试变红 → ✅ 测试在兜底
  ↓
没有变红 → ❌ 这条没被测住，回去补测试
```

---

## 6. 覆盖率门槛（CI 硬关）

> **没有门槛的覆盖率是自我安慰**。设个能强制执行的数。

| 指标 | 起步门槛 | 目标 |
|---|---|---|
| 行覆盖率 | 70% | 85%+ |
| 分支覆盖率 | 60% | 75%+ |
| 关键路径 | 100% | 100% |
| 新增代码 | 与存量持平或更高 | — |

> **降覆盖率需 PR 显式说明**（"删了未用代码" / "合并到已有测试"）。

## 7. 提示词模板（直接复制用）

**红**：
> 按这份 spec，先**只写测试**，不要写实现。spec §5 边界小节每条写一条测试。跑通后把输出给我看。

**绿**：
> 现在补实现，让上面的测试全绿。一个端点一个端点来。中途失败，把具体断言 + 输入 + 期望值列给我。

**重构**：
> 测试保持绿的前提下，清理实现（不改外部行为）。每改一处就跑一次测试。

**故意改坏自检**：
> 现在我要做一次自检：故意改坏一处实现（比如把空标题校验去掉），看对应测试是否会红。改完跑测试，把"哪条变红"列给我，然后恢复。

---

## 8. 进阶：把测试当合同

测试是**机器能判对错的合同**。spec 容易被"再讨论一下"，测试红了你没法装看不见。

```
spec（人对齐的意图）
  ──▶  单元 / 集成 / 契约 / E2E 测试（机器能判的合同）
  ──▶  实现
  ──▶  故意改坏验证 ◀──── 自检闭环
```

---

## 9. 反模式（看到就停）

| 反模式 | 后果 | 改成什么 |
|---|---|---|
| 全部写在 setUp 里 | 调试时不知哪个 fixture 出问题 | 显式注入依赖 |
| 测私有方法 | 重构即崩 | 测公有行为 |
| mock 返回固定值不验行为 | mock 错也绿 | assert_called_with + 状态断言 |
| 一个 it 测 10 件事 | 失败信息糊 | 一个 it 一个断言点 |
| 用 sleep 等异步 | 慢 + 偶发 | 用轮询 / 信号量 / 确定性 wait |
| 覆盖率 100% 为目标 | 写无聊的 getter 测试 | 关键路径 100%，其他 70%+ |
| 测试数据用真实邮箱/手机 | 污染 + 合规风险 | 用 `*@example.com` / 测试号段 |

---

## 一句话判断

> **你 review 的是测试，不是实现。** 审查成本被前移、压缩了。
> **你 CI 守的是测试，不是实现。** 跑偏一次，CI 拦住，PR 被打回。
